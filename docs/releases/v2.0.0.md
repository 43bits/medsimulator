# Version 2.0.0 release notes

Invinceible Core HMS Version 2 prepares the product for a production-grade
Render release with a cleaner fixed light interface, stronger deployment
documentation, and broader operational readiness across clinical, pharmacy,
billing, platform administration, and reporting workflows.

## Release highlights

- Render backend and frontend deployment support through `render.yaml`.
- Render PostgreSQL migration preparation with MySQL rollback support.
- Fixed professional light-theme UI with runtime theme switching removed.
- Improved official invoice, receipt, medical report, and SHA printout/PDF foundations.
- Improved consultation, prescribing, pharmacy dispensing, and stock safety workflows.
- Improved billing, invoices, payments, revenue integrity, and M-Pesa configuration.
- Expanded platform administration for facilities, branches, clinics, departments, staff, users, roles, notifications, subscriptions, and audit visibility.
- Strengthened authentication, password policy, production secret validation, rate limits, request timeouts, and audit redaction.
- Improved health checks, logging, observability, pagination, scoped search, cache settings, and queue configuration.

## UI cleanup

The frontend is now a fixed light-theme HMS interface. Unused dark mode/theme
switching code was removed to reduce hydration risk, remove UI ambiguity, and
keep clinical/admin screens visually stable for release.

## Deployment notes

Version 2 supports two production database modes:

1. Render application services using the existing external MySQL database.
2. Render application services using Render PostgreSQL after a tested migration.

Do not point production to Render PostgreSQL until the MySQL database has been
backed up, imported, validated, and rollback-tested. Follow
[MySQL to Render PostgreSQL migration](../deployment/mysql-to-render-postgres.md).

Required Render checks:

- Backend root: `backend`
- Frontend root: `frontend`
- Backend health check: `/health/live`
- Backend production start: `npm run start:prod`
- Frontend production start: `npm run start`
- Backend PostgreSQL build: `npm ci && npm run prisma:generate:postgres && npm run build`
- Backend PostgreSQL pre-deploy: `npm run prisma:migrate:postgres`
- Frontend build: `npm ci && npm run build`

## Upgrade checklist

- Configure `JWT_SECRET` with a high-entropy production value.
- Configure `DATABASE_PROVIDER=postgresql` only after the PostgreSQL migration is validated.
- Configure `DATABASE_URL` in Render, never in committed files.
- Configure `FRONTEND_URL` and `FRONTEND_ORIGINS`.
- Configure `NEXT_PUBLIC_API_BASE_URL` on the frontend.
- Verify M-Pesa callback URLs point to the final backend domain.
- Run backend and frontend production builds.
- Test login, patient flow, consultation, pharmacy, billing, reports, PDFs, and platform administration.
- Confirm no secrets are committed.

## Known limitations

- PostgreSQL production cutover still requires a real data migration rehearsal.
- Existing backend Jest specs need test-module provider cleanup before the full suite is reliable.
- AI, patient portal, SMS, WhatsApp, and data warehouse features remain controlled by feature flags and should stay disabled unless fully configured.
